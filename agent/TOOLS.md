# AI Engineering Auditor — Tools

Version: 0.1.0

---

## Purpose

This document defines the tools available to the AI Engineering Auditor Agent.

The Agent must use these tools as mechanical operations over the current repository.

The Agent reasons.
The CLI executes.

---

## Core rule

The user should not need to execute CLI commands manually during normal work.

The Agent should decide when a command is needed and invoke or suggest the correct command depending on the environment.

---

## Available CLI commands

### aiea-init

Initializes AI Engineering Auditor in the current repository.

Use when:

- the repository does not contain `docs/ai-auditor`
- the user starts auditing a new project
- the Agent detects that the Auditor is not initialized

Effect:

- creates `docs/ai-auditor`
- creates project files
- creates the initial `ProjectInitialized` event

The Agent should not ask the user to create files manually.

---

### aiea-status

Reads the current Auditor state.

Use when:

- starting a new Agent session
- checking whether the project is initialized
- checking current phase
- checking whether a task is active
- verifying event count

Effect:

- does not modify files
- prints project status

The Agent should use this before making assumptions about current state.

---

### aiea-task

Manages the current active task.

Supported commands:

```bash
aiea-task status
aiea-task start "Task title"
aiea-task close
Use when:

a user starts a new engineering task
a user finishes a task
the Agent needs to check task state

Important:

aiea-task only updates state.json
it does not create events
task events must be created separately using aiea-event
aiea-event

Creates a new event in events.jsonl.

Supported command:

aiea-event create

Input:

JSON payload through stdin

Example:

echo '{"eventType":"Finding","phase":"Discovery","tool":"Codex","title":"Realm uses embedded objects"}' | aiea-event create

Use when:

registering AI usage
registering findings
registering risks
registering decisions
registering issues
registering milestones
registering task activity

Important:

the Agent must generate the JSON payload
the CLI adds eventId and timestamp
all events must go through this command or the SDK
aiea-reset

Removes Auditor files from the current project.

Use only for:

development
testing
resetting a sample project

Do not use in a real audited project unless the user explicitly requests it.

Normal Agent startup flow

When the Agent starts inside VS Code:

Run or request aiea-status.
If the project is not initialized, initialize it with aiea-init.
Read current status.
Check whether a task is active.
Ask the user what they are working on only if no task is active.
Starting a task

When the user says something like:

"Voy a analizar Realm"
"Empiezo con el discovery"
"Voy a revisar la arquitectura"
"Voy a probar Codex con este módulo"

The Agent should:

Normalize the task title.
Run:
aiea-task start "<normalized task title>"
Register a TaskStarted event using aiea-event.

Example payload:

{
  "eventType": "TaskStarted",
  "phase": "Discovery",
  "tool": "Manual",
  "title": "Analyze Realm persistence",
  "description": "Task started by the developer."
}
Registering AI usage

When the user uses an AI tool during a task, the Agent should register an AIInteraction event.

Example payload:

{
  "eventType": "AIInteraction",
  "phase": "Discovery",
  "tool": "Codex",
  "title": "Codex analyzed Realm persistence",
  "description": "Used Codex to identify Realm models and persistence flows.",
  "purpose": "Analysis",
  "result": "Accepted",
  "accepted": true
}

The Agent should ask only for missing information.

Preferred questions:

Which tool was used?
Was the output accepted, modified, rejected or informational?
Was meaningful human correction required?
Closing a task

When the user says:

"He terminado"
"Cierra la tarea"
"Ya he revisado esto"
"Terminamos este análisis"

The Agent should:

Ask only for missing evaluation data.
Register relevant AIInteraction, Finding, Risk, Decision or Issue events.
Register a TaskCompleted event.
Run:
aiea-task close

Example TaskCompleted payload:

{
  "eventType": "TaskCompleted",
  "phase": "Discovery",
  "tool": "Manual",
  "title": "Analyze Realm persistence completed",
  "description": "Task completed and reviewed."
}
Event generation rules

The Agent must generate compact, valid JSON payloads.

Required fields:

eventType
phase
tool
title

Optional fields:

description
taskId
severity
purpose
result
accepted
metadata

The Agent must not generate eventId.

The Agent must not generate timestamp.

Those fields are generated by the CLI/SDK.

Tool responsibility boundaries

The Agent must not use CLI tools for reasoning.

The CLI does not decide:

task classification
AI usefulness
risk level
business meaning
final conclusions

The Agent decides those things and records them through events.

Golden rule

If something should appear in the final AI evaluation report, it must be captured as an event.
```
